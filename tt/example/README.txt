================================================================================
         سیستم نظارت و کنترل بلادرنگ گلخانه (TinyTimber)
================================================================================

1. معرفی پروژه
--------------------------------------------------------------------------------

این پروژه یک سیستم تعبیه‌شده بلادرنگ برای شبیه‌سازی گلخانه است که با
فریم‌ورک TinyTimber (دانشگاه لولئو) پیاده‌سازی شده است.

اهداف:
  - شبیه‌سازی سنسورهای مجازی (دما، رطوبت، نور)
  - کنترل عملگرهای مجازی (فن، پمپ آب، لامپ)
  - طراحی واکنش‌گرا (Reactive) مبتنی بر پیام‌های TinyTimber
  - تحلیل زمان‌بندی بلادرنگ
  - اجرای سناریوهای آزمایش و اعتبارسنجی

تمام کد برنامه در یک فایل قرار دارد:
  tt/example/controller/src/main.c

هسته TinyTimber (pT/ و env/) تغییر داده نشده است.


2. معماری سیستم
--------------------------------------------------------------------------------

هر بخش یک «شیء واکنش‌گر» TinyTimber است. اولین فیلد هر ساختار:
  tt_object_t object

جریان داده:

  سنسور دما      --TT_ASYNC-->  کنترلر  --TT_ASYNC-->  فن
  سنسور رطوبت    --TT_ASYNC-->      |    --TT_ASYNC-->  پمپ
  سنسور نور      --TT_ASYNC-->      +-- TT_AFTER(100ms) controller_tick

راه‌اندازی (مانند نمونه استاد):
  tt_init()  ->  init()  ->  tt_run()

پیام‌های اصلی:
  TT_AFTER   : کار دوره‌ای (سنسورها و کنترلر)
  TT_ASYNC   : ارسال داده یا فرمان به شیء دیگر
  TT_ARGS_NONE : بدون آرگومان


3. توضیح سنسورها
--------------------------------------------------------------------------------

3.1 سنسور دما
  - نوع: float (درجه سانتی‌گراد)
  - دوره: هر 200 میلی‌ثانیه (PERIOD_TEMP)
  - مقدار پایه: حدود 28 درجه (بازه 24 تا 32)
  - نویز: ±0.5 درجه (حالت عادی)
  - در سناریو نویز شدید: ±2.0 درجه
  - متد: temp_sensor_sample

3.2 سنسور رطوبت
  - نوع: درصد (float)
  - دوره: هر 500 میلی‌ثانیه (PERIOD_HUM)
  - مقدار: تصادفی بین 20 تا 80 درصد
  - متد: hum_sensor_sample

3.3 سنسور نور
  - نوع: عدد صحیح (0 تا 100)
  - نوع رویداد: Sporadic (Sporadic Task)
  - فاصله تصادفی: 1 تا 4 ثانیه
  - متد: light_sensor_event


4. توضیح کنترلر
--------------------------------------------------------------------------------

کنترلر هر 100 میلی‌ثانیه اجرا می‌شود (PERIOD_CONTROLLER).

قوانین کنترل:

  اگر دما > 28 درجه         ->  فن = ON
  اگر رطوبت < 35 درصد       ->  پمپ = ON
  اگر نور < 40 (آستانه)     ->  لامپ = ON
  در غیر این صورت           ->  OFF

نمونه لاگ:
  Controller: Temp=30.7 -> Fan ON

متد اصلی: controller_tick


5. توضیح عملگرها (Actuators)
--------------------------------------------------------------------------------

هر عملگر یک state machine ساده با دو حالت OFF و ON دارد.

  - fan_t   : فن
  - pump_t  : پمپ آب
  - lamp_t  : لامپ

تغییر حالت فقط با TT_ASYNC انجام می‌شود.
اگر حالت قبلی و جدید یکسان باشد، لاگ چاپ نمی‌شود.

نمونه لاگ:
  FAN -> ON
  FAN -> OFF


6. توضیح سناریوهای آزمایش
--------------------------------------------------------------------------------

برنامه 12 ثانیه اجرا می‌شود و سپس خاتمه می‌یابد.

سناریو 1 - افزایش ناگهانی دما (temp_spike)
  - پس از 3 ثانیه دمای پایه به 31 درجه افزایش می‌یابد
  - بررسی: آیا فن روشن می‌شود؟

سناریو 2 - نویز شدید سنسور (severe_noise)
  - دامنه نویز دما به ±2.0 درجه افزایش می‌یابد
  - بررسی: پایداری سیستم

سناریو 3 - بار سنگین (heavy_load)
  - حلقه اضافی در هر handler سنسور
  - یک پیام دوره‌ای هر 50ms برای افزایش بار
  - بررسی: پاسخ سیستم تحت بار

سناریو 4 - خرابی عملگر (fan_failure)
  - فن در حالت خراب گیر می‌کند (failed=1)
  - دما بالا می‌رود
  - بررسی: ثبت لاگ خطا


7. توضیح لاگ‌ها
--------------------------------------------------------------------------------

فرمت:
  [  زمان ms][سطح][اجزاء     ] پیام

سطوح:
  INFO : رویداد عادی
  WARN : هشدار (سناریو، خرابی فن)
  MISS : از دست رفتن deadline (در صورت رخداد)

رویدادهای ثبت‌شده:
  - نمونه‌برداری دما و رطوبت
  - رویدادهای نور
  - تصمیم کنترلر
  - تغییر حالت عملگرها
  - شروع/پایان سناریو

--------------------------------------------------------------------------------

کامپایل:

9. تحلیل بلادرنگ
--------------------------------------------------------------------------------

9.1 جدول وظایف (Task Table)
--------------------------------------------------------------------------------

+------------------------+--------+----------+-------------+----------+
| نام وظیفه              | دوره   | اولویت   | WCET تخمینی | Deadline |
+------------------------+--------+----------+-------------+----------+
| controller_tick        | 100 ms | 1 (بالا) | 0.15 ms     | 100 ms   |
| temp_sensor_sample     | 200 ms | 2        | 0.08 ms     | 200 ms   |
| hum_sensor_sample      | 500 ms | 3        | 0.08 ms     | 500 ms   |
| light_sensor_event     | Sporadic (MIT>=1s) | 4 | 0.06 ms | 500 ms |
+------------------------+--------+----------+-------------+----------+

9.2 تخصیص اولویت
--------------------------------------------------------------------------------

اولویت‌ها بر اساس Rate Monotonic (RM) تعیین شده‌اند:
  - دوره کوتاه‌تر -> اولویت بالاتر
  - کنترلر (100ms) بالاترین اولویت منطقی
  - سنسور نور sporadic با اولویت پایین‌تر

در TinyTimber POSIX زمان‌بندی بر اساس baseline و deadline پیام است.


9.3 تخمین WCET
--------------------------------------------------------------------------------

WCETها برای شبیه‌سازی POSIX و کد ساده undergraduate تخمین زده شده:

  controller_tick     : 0.15 ms  (چند مقایسه + sprintf + TT_ASYNC)
  temp_sensor_sample  : 0.08 ms  (rand + log + TT_AFTER)
  hum_sensor_sample   : 0.08 ms
  light_sensor_event  : 0.06 ms

در سناریو heavy_load WCET سنسورها بیشتر می‌شود (حلقه spin).


9.4 آزمون بهره‌وری Rate Monotonic
--------------------------------------------------------------------------------

فرمول:
  U = sum (Ci / Ti)

محاسبه (حالت عادی):
  U = 0.15/100 + 0.08/200 + 0.08/500 + 0.06/1000
    = 0.0015 + 0.0004 + 0.00016 + 0.00006
    = 0.00212

مرز RM برای n=4:
  U_max = 4 * (2^(1/4) - 1) = 0.756

نتیجه:
  U = 0.002 << 0.756  ->  سیستم از نظر RM schedulable است.


9.5 تحلیل زمان پاسخ (Response Time Analysis)
--------------------------------------------------------------------------------

برای کنترلر (T=100ms, C=0.15ms, اولویت بالا):

  R0 = C = 0.15 ms

  تداخل از وظایف با دوره کوتاه‌تر یا هم‌سطح:
  - temp: every 200ms, C=0.08
  - hum:  every 500ms, C=0.08
  - light: sporadic, C=0.06

  در یک پنجره 100ms حداکثر یک اجرای temp ممکن است:
  R1 = 0.15 + 0.08 = 0.23 ms

  R2 = 0.23 + 0.08 = 0.31 ms (humidity)

  R = 0.31 ms << D = 100 ms

نتیجه: deadline کنترلر رعایت می‌شود.


9.6 نتایج اعتبارسنجی
--------------------------------------------------------------------------------

سوال: آیا همه deadlineها رعایت شد؟
  - در شبیه‌سازی بله؛ WCETها بسیار کوچک و U بسیار کم است.
  - در heavy_load بار CPU بیشتر می‌شود ولی هنوز schedulable است.

سوال: آیا سیستم به‌موقع پاسخ داد؟
  - temp_spike: فن ظرف یک دوره کنترلر (100ms) فرمان ON می‌گیرد.
  - fan_failure: فرمان ON صادر می‌شود ولی فن stuck است -> WARN ثبت می‌شود.

سوال: آیا رفتار پایدار و درست بود؟
  - normal: کنترل بر اساس قوانین
  - severe_noise: نوسان بیشتر ولی کنترل پایدار
  - heavy_load: سیستم تحت فشار ولی بدون crash
  - fan_failure: خطا به‌درستی log می‌شود


10. جمع‌بندی
--------------------------------------------------------------------------------

این پروژه با حفظ ساختار نمونه استاد (tt/example/controller) و استفاده
از API رسمی TinyTimber، یک گلخانه مجازی کامل با سنسور، کنترلر، عملگر،
لاگ و چهار سناریو آزمایش ارائه می‌دهد.

تحلیل RM و RTA نشان می‌دهد سیستم schedulable است و deadline کنترلر
با حاشیه امن زیاد رعایت می‌شود.
